# AstroGraph 项目完整落地待办清单

> 目标：在**功能完整**、**性能可支撑大规模数据（如 75GB 级图库）**、**体验流畅**的前提下，将当前系统推进到可长期维护的生产形态。  
> 说明：下列条目按**依赖关系**与**优先级**组织；同一阶段内可按并行任务拆分给多人。

## 优先级总览（P0 → P3，建议严格按序推进）

> **原则**：先「数据能进、图能显、服务能测」，再「搜得准、答得好」，最后「企业级治理与持续运营」。下表与后文「阶段 0～8」「第十五～十九章」一一对应。

| 优先级 | 含义 | 对应章节（本文） | 解决的核心问题 |
|--------|------|------------------|----------------|
| **P0** | 必须先完成，否则后续无法做真效果 | **第十五章**（数据集落盘与 `CSV_ROOT`）+ **二、阶段 0** + **三、阶段 1** 中 3.1～3.2 的主体 | 数据从哪来、怎么进系统、图片/元数据是否可追溯；环境可复现、可压测 |
| **P1** | 决定「大规模 + 准检索」 | **三、阶段 1**（3.3 图谱）+ **四、阶段 2** 全文 | 76GB 级图库不拖垮服务；语义级以文搜图/以图搜图；可评测、可调参 |
| **P2** | 决定「智能问答与产品体验」 | **五、阶段 3** + **六、阶段 4** + **七、阶段 5** | GraphRAG 可引用、可核验；API/安全/限流；前端流畅、图谱可交互 |
| **P3** | 决定「企业级可上线、可运维」 | **八～十**（阶段 6～8）+ **第十六章** + **第十七章**（创新落地项） | 可观测、可告警、可备份恢复；CI/CD、多环境；审计与合规；持续迭代 |

**并行提示**：P0 内「环境/数据集」可与「代码分支」并行；P1 内「Milvus」与「Embedding 批处理」可并行；P2 内「后端安全」与「前端性能」可并行。**不要**在 P0 未完成时深度投入 P2 的 LLM 调参，否则易返工。

**读完优先级后**：直接看 **第二十章**（全部完成后的项目画像）；执行细节仍按 **二～十九章** 逐项勾选。

## 一、范围与验收口径

### 1.1 产品能力范围（需全部覆盖）

| 域 | 能力 | 验收要点 |
|---|------|----------|
| 账号与安全 | 注册、登录、登出、会话、权限 | 未登录不可访问业务页；Token 失效可感知；密码安全存储 |
| 多模态检索 | 以文搜图、以图搜图、混合条件 | Top-K 稳定、可解释、可溯源 |
| 知识图谱 | 实体浏览、路径发现、对比、2D/3D 可视化 | 大图下仍可交互；空数据有引导 |
| 智能问答 | GraphRAG/检索增强问答 | 答案含引用；引用可点开核验 |
| 数据工程 | 接入、清洗、建索引、增量更新 | 可重复执行；失败可恢复；可观测 |
| 运维与性能 | 监控、限流、缓存、压测、扩容路径 | 有 SLO；有告警；有压测报告 |

### 1.2 非功能指标（建议写进里程碑）

- **可用性**：核心 API 月可用性 ≥ 99.5%（单机可先 ≥ 99%）。
- **延迟**：P95 检索 < 800ms（10 万级索引样本下）；P95 问答 < 3s（依赖 LLM 时可拆分异步）。
- **吞吐**：在目标 QPS 下 CPU/内存不持续打满；无内存泄漏。
- **数据规模**：百万级元数据记录可分页浏览；亿级向量需分片/分区策略明确。

## 三、阶段 1：数据与存储底座（核心，阻塞后续）

### 3.1 元数据与文件规范

- [ ] **统一 `image_id` 规范**：全链路（上传、索引、URL、引用）禁止裸路径。

## 四、阶段 2：向量检索生产化（效果与性能的关键）

### 4.1 Embedding 与模型

- [ ] **文本向量模型选型**：BGE / E5 等，确定维度与语言支持。
- [ ] **图像向量模型选型**：CLIP / SigLIP 等，确定输入尺寸与预处理。
- [ ] **模型部署**：本机推理 / TorchServe / Triton（按团队能力选）。
- [ ] **批处理推理**：GPU 利用率、batch size、队列背压。

### 4.2 Milvus（或等价向量库）

- [ ] **Collection 设计**：分区键（时间/类别/数据源）、字段 schema。
- [ ] **索引类型**：HNSW/IVF 选择与参数调优。
- [ ] **插入与Compaction**：批量 insert、flush、compaction 策略。
- [ ] **查询路径**：过滤 + TopK + rerank（交叉编码器可选）。
- [ ] **混合检索**：文本 + 图像 hint 的融合策略（RRF/加权/学习排序后置）。

> **当前代码库（P1 高配路径，待环境就绪）**：OpenCLIP（ViT-B-32）+ Milvus 集合（默认 `astro_image_clip`，可由 `MILVUS_COLLECTION` 覆盖）；`backend/scripts/index_milvus_clip.py` 对已加载的 `images_catalog` 实体按本地 `ref` 路径编码入库；接口 `GET/POST /api/v1/image/search-by-text|search-by-image` 支持 `page`/`page_size` 分页；`GET /api/v1/image/vector-status` 用于探针。**知识图谱**构建与可视化已排除 `images_catalog.csv`，节点来自其余 info CSV。本地需：Milvus 可连、`.env` 中 `MILVUS_ENABLED=true`、数据加载后跑索引脚本。

### 4.3 召回质量与评测

- [ ] **评测集**：构造 query-相关图片/文档标注（哪怕小规模）。
- [ ] **指标**：Recall@K、MRR、nDCG；线上抽样人工评估流程。
- [ ] **Badcase 闭环**：记录失败 query，驱动迭代。

## 十五、数据集下载、存放与接入（建议最先完成）

> 典型结构：`data/images/{train,test}/...` + `data/info/*.csv`（约 76GB 级图库）。与本项目后端约定：**`CSV_ROOT` 指向 `info` 目录**（仅该层会被 `*.csv` 扫描）。

### 15.1 下载与校验

- [ ] **来源**：从官方渠道（如 Kaggle / 竞赛平台）下载；保留页面上的**许可协议**与引用说明，写入内部 `DATASET_LICENSE.md`（不入库亦可，团队知识库必存）。
- [ ] **完整性**：大文件优先校验 **SHA256/MD5**（若平台提供）；分卷压缩包核对分卷数量与解压无报错。
- [ ] **空间**：预留 **数据集体积 × 1.3～1.5** 磁盘（解压、缩略图、向量索引、临时文件）。
- [ ] **解压路径**：使用**纯英文路径**（避免部分工具编码问题），例如 `D:\datasets\AstroImage\data\`。

### 15.2 落盘位置（不要塞进 Git）

- [ ] **禁止**将整包数据提交到仓库；项目根 `.gitignore` 增加 `dataset/`、`data/` 或你实际使用的数据目录名。
- [ ] **推荐**：数据与代码分离，例如 `D:\datasets\...\data\`，后端通过环境变量指向 `...\data\info`。
- [ ] **配置**：在 `backend/.env` 设置 `CSV_ROOT=<绝对路径>/data/info`；确认进程用户对该路径及 `data/images` 有读权限。

### 15.3 接入前检查（避免加载后“无图”）

- [ ] 打开任意 `info/*.csv`，确认是否包含**图片路径或文件名列**（与 [`DataService`](../backend/app/services/data_service.py) 中 `image_path` / `path` / `url` 等候选键一致或可映射）。
- [ ] 若 CSV **仅有文字、无逐图路径**：须走**离线扫描** `images/` 生成清单或与实体表关联（见阶段 3.2「增量扫描 / 批处理」），否则系统无法自动关联 76GB 图片。
- [ ] **train / test**：明确产品是否只索引 `train`、或两者都索引并在元数据加 `split` 字段，避免评测与展示混淆。

### 15.4 接入后验收（最小闭环）

- [ ] `GET /api/v1/data/status`：`entity_count`、`image_count` 与预期量级一致。
- [ ] 抽样调用图片文件接口，确认可打开；检索接口能返回带 `image_url` 的结果。

### 15.5 已落地基线（当前仓库）

- [x] 已支持通过 `backend/.env` 设置 `CSV_ROOT=.../data/info` 自动加载 CSV。
- [x] 已提供图片清单脚本：`backend/scripts/build_images_catalog_csv.py`（将 `images/train|test/...` 生成 `info/images_catalog.csv`）。
- [x] 已修复本地图片路径解析时序问题（加载过程中即可解析 `image_path`），避免 `image_count=0` 的假成功状态。

**建议命令（你的数据集路径）**：

```bash
cd backend
.venv\Scripts\python scripts\build_images_catalog_csv.py --dataset-root "E:\astronomical_dataset\astronomical-image-and-csv-dataset\data" --splits train
```

然后重载数据：

```bash
POST /api/v1/data/load
{
  "csv_root": "E:/astronomical_dataset/astronomical-image-and-csv-dataset/data/info",
  "categories": []
}
```

## 十六、企业级能力建设（目标与做法）

> “企业级”在此定义为：**可治理、可审计、可扩展、可恢复、可度量**，而非仅功能堆叠。

### 16.1 安全与合规

- [ ] **身份体系演进**：由演示 Token → **JWT（access + refresh）**、刷新撤销（Redis 黑名单或版本号）。
- [ ] **权限模型（按产品需要）**：角色（访客/用户/管理员）、后台操作与敏感接口隔离。
- [ ] **审计日志**：登录、改密、建索引、删数据等关键操作可追溯（谁、何时、何对象）。
- [ ] **密钥管理**：生产环境密钥来自环境变量/密钥管理服务，**禁止**提交 `.env` 真值。
- [ ] **隐私与外链**：用户数据、外链跳转、图片引用符合团队合规与版权声明。

### 16.2 可靠性与灾备

- [ ] **SLA/SLO 成文**：核心 API 可用性、P95 延迟、错误率预算；与监控告警阈值对齐。
- [ ] **备份策略**：DB、对象存储、向量索引元数据；明确 **RPO/RTO**（能丢多久数据、多久恢复服务）。
- [ ] **恢复演练**：季度至少一次“从备份拉起”演练，避免灾难时手册失效。
- [ ] **多实例路径（可选）**：无状态 API + 共享存储/会话外置，为横向扩容留接口。

### 16.3 API 与数据治理

- [ ] **API 版本化**：`/api/v1` 冻结规则；破坏性变更走 v2 或兼容层。
- [ ] **幂等与重试**：上传、建索引、批任务带幂等键；客户端可安全重试。
- [ ] **数据血缘**：每条索引记录可追溯：来源文件、版本、构建任务 ID。
- [ ] **质量门禁**：入库前规则校验（缺字段、非法路径、重复率），异常进入隔离区人工审核。

### 16.4 工程化与发布

- [ ] **CI/CD**：合并前 lint/test/build；主干保护；发布标签与变更日志。
- [ ] **灰度与回滚**：新版本先小流量或预发环境验证；一键回滚到上一镜像/配置。
- [ ] **配置分层**：dev / staging / prod 配置分离；特性开关（Feature Flag）控制新检索策略等。

## 二十、若本清单全部实现：你将得到什么样的项目

> 下列描述假设 **P0～P3 与清单内勾选项均落实**（含真实向量库、对象存储、图谱库、LLM 与运维体系）。若只完成部分优先级，能力会按 **20.5 小节** 的表格递减。

### 20.1 项目定位（你拥有的是什么）

- 一个 **天文领域多模态知识探索平台**：以 **大规模图像 + 结构化实体/关系 + 向量检索 + 图谱推理 + 大模型生成** 串联，面向科普展示、教学演示或轻量科研辅助。
- 技术上属于 **RAG + Graph + CV** 的组合形态，具备 **可替换模型、可扩展数据管道、可观测运维** 的企业级骨架，而非单一「演示 Demo」。

### 20.2 用户能做什么（功能层面）

| 能力域 | 用户侧表现（功能） |
|--------|-------------------|
| 账号 | 注册登录、会话保持、（按实现）角色与权限、个人历史与基础资料 |
| 图像 | 以文搜图、以图搜图、结果可分页/可筛选（按实现）、缩略图快速浏览、点击溯源到实体或来源 |
| 知识图谱 | 实体检索、多跳路径、实体对比、2D 关系图加载与交互；（按实现）与类别/数据源联动 |
| 智能问答 | 自然语言提问；答案带 **引用与推理链提示**；可结合图谱子图与检索片段 |
| 数据与运维（管理员/后台） | 数据接入任务可监控、可重试；索引与构图任务有状态；系统健康与指标可查 |

### 20.3 效果与体验（「能做到多好」）

- **检索效果**：由「关键词重合」升级为 **语义级召回 + 可选重排**；在合理标注与调参下，**相关图片/文档的命中率**可客观用 Recall@K、线上抽样评估度量，而非凭感觉。
- **响应与流畅度**：在文档 **1.2** 与 **18.1** 的目标下，核心检索路径 **P95 延迟可控**；大图库通过 **缩略图、懒加载、虚拟列表、CDN（可选）** 保持前端滚动与首屏流畅；重任务 **异步化**，避免接口卡死。
- **问答可信度**：GraphRAG 完整落地后，答案强调 **可引用、可点开核验**；幻觉风险通过 **检索约束 + 引用校验** 显著低于「纯模型瞎答」。
- **规模**：**百万级**元数据浏览、**亿级向量**分区扩展路径清晰；76GB 级图库通过 **对象存储 + 分层索引** 纳入常规运维，而非「单机文件夹硬撑」。

### 20.4 工程与组织价值（企业级「能交付什么」）

- **可交付**：多环境配置、CI/CD、镜像与回滚、备份与演练——满足「交给运维/甲方验收」的最低门槛。
- **可迭代**：Badcase 与评测闭环、特性开关、API 版本策略——新模型/新数据可 **灰度** 上线。
- **可合规**：审计日志、密钥管理、数据血缘与质量门禁——便于对内汇报或对外说明数据来源与使用边界。

### 20.5 与「只做完一部分」的差别（心理预期）

| 完成边界 | 你得到的大致形态 |
|----------|------------------|
| 仅 P0 | 数据可加载、服务可测、**效果偏规则/占位检索**，适合联调与数据验证 |
| P0 + P1 | **真实语义检索 + 大规模可扩展**，产品「搜图」核心成立 |
| P0～P2 | **智能问答 + 完整前后端体验**，可对外演示「像产品的系统」 |
| P0～P3 + 清单全部 | **可长期运行、可运维、可演进的生产级系统**（仍受业务预算与模型选型影响上限） |
