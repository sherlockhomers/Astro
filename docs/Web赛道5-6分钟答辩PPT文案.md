## 第2页 项目背景

### 页面标题
项目背景

### 页面中心句
当前天文科普内容很多，但真正好用、好懂、好交互的 Web 平台并不多。

### 页面上屏文字
- 天文知识来源分散，检索成本高
- 传统科普网站多为静态图文，交互深度不足
- 普通用户面对专业概念时理解门槛较高

### 页面补充句
现有多数科普网站以静态展示为主，缺少“提问-检索-关联-可视化探索”的闭环。

### 配图建议
- 左侧放 3 个痛点图标。
- 右侧放“传统静态展示”与“交互式探索”对比示意图。
- 这一页不必点名具体竞品，避免把篇幅带偏。

### 口播提示
我们在调研中发现，天文科普并不缺内容，缺的是面向普通用户的低门槛交互方式。很多平台可以“看”，但很难做到“问了以后还能继续查、继续比、继续探索”。这正是我们做这个项目的出发点。

### 建议时长
25-30秒

## 第5页 功能演示链路

### 页面标题
功能演示链路

### 页面中心句
系统把一次问答，延伸为一次可视化探索过程。

### 页面上屏流程
首页进入  
-> 输入问题“木星为什么会有这么多卫星？”  
-> 智能问答返回解释  
-> 图谱延展查看关联关系  
-> 图像检索查看观测样本  
-> 3D模型查看木星形态

### 页面强调句

### 配图建议
- 用横向流程箭头。
- 配 4 张截图：Landing、QA、Knowledge、Starfield。
- 问题示例固定使用“木星为什么会有这么多卫星？”，避免现场口径不一致。

### 口播提示
这一页是我们答辩时最重要的一页。我们会用一个具体问题串起整套系统：用户从首页进入后提问，系统先给出自然语言解释，再提供图谱关系延展，之后还能继续看图像样本和三维模型。也就是说，这个平台不是“答完就结束”，而是“答完之后还能继续探索”。

## 第9页 技术架构与创新点

### 页面标题
技术架构与创新点

### 页面中心句
技术不是堆叠，而是服务于更自然、更稳定的天文科普体验。

### 页面上屏架构
前端交互层  
-> 首页门户、问答页面、知识图谱、图像检索、3D展示

后端服务层  
-> 用户服务、问答服务、检索服务、图谱服务、评测服务

智能能力层  
-> 智能体协同问答链路、GraphRAG、混合检索、多模态检索

数据层  
-> 结构化数据、文本语料、图像索引、图谱数据

### 页面上屏创新点
- 智能体协同问答链路，提升回答稳定性
- GraphRAG + 知识图谱，支持关联解释与路径推理
- 图谱、图像、3D 融合的多模态探索体验

### 配图建议
- 左半部分放四层架构图。
- 右半部分只放 3 条创新点。
- 这一页不要展开算法细节，重在“技术为什么有用”。

### 口播提示
在技术实现上，我们没有把重点放在单一模型上，而是放在“链路设计”上。系统会根据问题类型调用检索、图谱和问答模块协同工作，从而让答案更稳定、解释更完整、探索路径更自然。这也是我们认为它适合 Web 赛道的核心原因。

## 附录三：答辩时不要说的话
