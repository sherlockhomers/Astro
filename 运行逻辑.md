# 天文科普问答系统模块设计与工作逻辑

请帮我按照以下模块划分和工作逻辑来构建代码架构：

## 模块一：多模态检索模块（独立的图片搜索引擎）
这个模块完全独立运行，专注于图片搜索功能。实现以下逻辑：
1. 将5万张天文图片通过CLIP模型编码为向量，存入Milvus数据库
2. 提供两个核心搜索接口：
   - 以文搜图：接收文字描述，返回最相似的天文图片
   - 以图搜图：接收上传的图片，返回最相似的天文图片
3. 这个模块有自己的API端点，可以被其他模块调用，也可以独立使用
4. 搜索时返回图片URL、相似度分数、简短描述

## 模块二：智能问答系统（核心模块）
这是系统的主要问答引擎，包含以下子模块：

### 2.1 知识图谱构建模块
从Excel文件中读取945条天文数据，构建Neo4j知识图谱。包含以下逻辑：
- 从Excel解析天体信息，包括实体（恒星、行星、星系等）和关系
- 构建图节点，包括天体属性、星座归属、发现信息等
- 建立图关系，如"属于星座"、"围绕旋转"、"被发现"等
- 将结构化数据存入Neo4j图数据库

### 2.2 GraphRAG检索模块
实现知识图谱增强检索，包含以下逻辑：
- 接收用户问题，提取问题中的关键天文实体
- 将自然语言问题转换为Cypher查询语句
- 在知识图谱中执行多跳查询（1-3跳关系）
- 将图查询结果转换为自然语言描述，作为回答的上下文
- 当知识图谱中信息不足时，触发补充策略

### 2.3 星语模型Agent（核心决策者）
这是系统的智能大脑，实现以下逻辑：
1. **意图分析**：分析用户问题类型，判断需要什么信息
2. **工具调度**：根据问题类型调用不同工具：
   - 基础知识问题 → 调用GraphRAG查询知识图谱
   - 最新信息问题 → 调用MCP工具获取实时数据
   - 需要图片的问题 → 调用多模态检索模块
3. **信息整合**：将多个来源的信息整合为统一格式
4. **流畅生成**：生成科普风格的连贯回答，避免字段堆砌

### 2.4 提示词工程模块
专门管理各种提示词模板，确保回答流畅：
- 通用问答模板：用于大多数问题
- 科普文章模板：用于需要详细解释的问题
- 比较分析模板：用于比较不同天体
- 图片描述模板：用于与图片相关的回答
- 每个模板都要求"先结论，再解释，后总结"的结构
- 用生活化例子做类比，将数字转化为易懂概念

## 模块三：扩展功能模块

### 3.1 MCP工具模块
实现标准化的工具接口，让Agent能获取实时信息：
- NASA APOD工具：获取每日天文图片
- 天象日历工具：获取近期天文事件
- 行星位置工具：获取实时行星位置
- 天文新闻工具：搜索最新天文发现
- 每个工具返回结构化数据，Agent可以调用

### 3.2 图片识别与问答模块
处理用户上传图片的场景：
1. **图片识别**：用视觉模型识别图片中的天体类型和名称
2. **问答流程**：结合识别结果和用户问题，进行智能问答
3. **混合处理**：当用户同时上传图片和文字问题时，综合处理

### 4.1 请求路由层

- 管理异步任务和超时

## 核心工作流程实现

请实现以下关键工作流：

### 工作流1：基础问答流程
用户输入文本问题 → Agent分析意图 → 调用GraphRAG查询知识图谱 → 整合信息 → 生成流畅回答

### 工作流2：知识补充流程
当知识图谱信息不足时 → Agent判断问题类型 → 如果是时效性问题 → 调用MCP工具 → 获取最新数据 → 整合回答

### 工作流3：图片搜索流程
用户上传图片或输入图片描述 → 路由到多模态检索模块 → 在Milvus中搜索相似图片 → 返回图片结果

### 工作流4：图片识别问答流程
用户上传图片并提问 → 图片识别模块识别天体 → 将识别结果传给Agent → Agent基于识别结果进行问答

## 关键技术点实现

1. **模块间通信**：使用REST API，每个模块提供清晰的接口
2. **错误处理**：一个模块失败不影响其他模块，有降级策略
3. **性能优化**：对高频查询做缓存，对大量数据做分批处理
4. **可扩展性**：可以轻松添加新的工具或数据源
